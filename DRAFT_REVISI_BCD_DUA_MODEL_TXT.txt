DRAFT REVISI B-C-D (FOKUS 2 MODEL: INDOBERT_NER + INDOBERT_REVISION_MATCHER)
Sumber disusun dari implementasi kode proyek (src/config.py, src/data_processing/*.py, src/training/*.py).

====================================================================
B. NORMALISASI DAN STANDARISASI (VERSI REAL IMPLEMENTASI)
====================================================================
Pada implementasi ini, normalisasi tidak dilakukan melalui lowercasing global berbasis aturan manual, tetapi melalui pipeline pra-anotasi dan pasca-pemrosesan yang terstruktur. Data mentah chat pada format task JSON diproses terlebih dahulu dengan pra-anotasi otomatis berbasis Regular Expression untuk mendeteksi entitas operasional awal (misalnya DATE, UNIT_QTY, UNIT_TYPE, TIME, ORIGIN, DESTINATION, DRIVER, PLATE, dan PHONE). Hasil pra-anotasi tersebut diimpor ke Label Studio untuk validasi manual, kemudian diekspor kembali ke format JSON.

Setelah anotasi manual, sistem melakukan konversi span karakter ke token subword menggunakan tokenizer indolem/indobert-base-uncased. Pada tahap ini, offset karakter dari Label Studio dipetakan ke token IndoBERT, lalu ditransformasikan ke skema label BIO (B-*, I-*, O). Tahap pembersihan komputasional berikutnya (post-processing cleaner) menerapkan aturan koreksi spesifik domain, yaitu: (1) pemotongan token nomor telepon yang melebihi 13 digit menjadi 12 digit pertama, (2) koreksi salah label token numerik berawalan "08" dari DATE menjadi PHONE, dan (3) penghapusan duplikasi suku kata/tokoh nempel (stuttering), misalnya "DediDedi" menjadi "Dedi". Dengan demikian, standarisasi dilakukan melalui kontrol kualitas entitas, bukan semata normalisasi huruf.

====================================================================
C. AUGMENTASI DATA (VERSI REAL IMPLEMENTASI)
====================================================================
Augmentasi pada proyek ini bersifat template-based paraphrase dan diterapkan pada dataset NER yang sudah bersih. Algoritma mengganti pemisah struktural ":" yang muncul tepat sebelum token entitas (B-LABEL) dengan frasa natural yang semantik-nya setara, misalnya untuk ORIGIN dapat berubah menjadi "muat di", "dari", atau "ambil di". Probabilitas penggantian diatur sebesar 50% per kandidat token yang memenuhi syarat.

Secara operasional, untuk setiap satu sampel asli dibuat dua sampel variasi tambahan, kemudian digabungkan kembali dengan data asli sehingga ukuran dataset menjadi tiga kali lipat (1x asli + 2x augmented). Seluruh sampel kemudian diacak (shuffle) agar distribusi pola bahasa lebih merata saat pelatihan model NER. Catatan penting: pada konfigurasi default saat ini, pelatihan NER membaca file data_siap_training_CLEAN.json; file augmentasi akan digunakan hanya jika path training diarahkan ke data_augmented.json.

Untuk model IndoBERT Revision Matcher, variasi bahasa juga dibentuk secara programatik pada saat konstruksi pasangan data, yaitu melalui synthetic revision text (contoh awalan "REQUEST ULANG ORDER ONCALL ...") yang dipasangkan dengan representasi kandidat terstruktur.

====================================================================
D. ANOTASI LABEL STUDIO DAN PEMBENTUKAN DATA LATIH DUA MODEL
====================================================================
Untuk arsitektur Named Entity Recognition (Tahap 1), data pra-anotasi divalidasi manual di Label Studio dan diekspor dalam format JSON-MIN. Hasil anotasi kemudian dikonversi ke token-level BIO. Kelas entitas operasional utama yang digunakan pada training NER adalah: UNIT_QTY, UNIT_TYPE, DATE, TIME, ORIGIN, DESTINATION, DRIVER, PLATE, dan PHONE (dengan kemungkinan entitas tambahan REASON pada konteks tertentu, namun fokus model operasional utama tetap pada sembilan entitas tersebut).

Untuk arsitektur Sequence-Pair Classification pada IndoBERT Revision Matcher (Tahap 2), implementasi saat ini tidak mengandalkan anotasi pasangan teks manual di Label Studio. Dataset pasangan dibentuk otomatis dari output NER yang sudah bersih melalui proses konstruksi kandidat dan pasangan semantik. Sistem menghasilkan pasangan positif (MATCH) dari teks asli maupun teks sintetik, serta pasangan negatif (NO_MATCH) melalui hard negative dan random negative sampling. Dengan demikian, pembelajaran relasi antar-pesan pada Revision Matcher ditopang oleh pairing otomatis terkontrol, bukan labeling manual pair-by-pair di antarmuka anotasi.

CATATAN FAKTA WAJIB UNTUK GEMINI (JANGAN DIUBAH)
====================================================================
1) Fokus model penelitian: IndoBERT NER + IndoBERT Revision Matcher.
2) Event classifier tidak dijadikan komponen inti dalam narasi dua model ini.
3) Data path baru:
   - Raw NER: data/chat/raw/data_mentah_NER2.json
   - Prelabel import: data/chat/raw/data_siap_import_NER2.json
   - Export Label Studio: data/chat/raw/export_label_studio_NER.json
   - Processed NER clean: data/chat/processed/NER/data_siap_training_CLEAN.json
   - Dataset Revision Matcher: data/chat/processed/SPC/revision_matcher_dataset.json
4) Label Revision Matcher: MATCH dan NO_MATCH.
5) Augmentasi NER bersifat opsional terhadap training default.
